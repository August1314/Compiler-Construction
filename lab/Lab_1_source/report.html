1. 实验目标

在原始实验软装置基础上，完成可提交、可复现、可测试的 Postfix 转换程序，并满足评分项对规范性与鲁棒性的要求。

  1. 将 Parser.lookahead 从静态成员改为实例成员，消除共享状态风险。
  2. 消除 rest() 尾递归，用循环等价重写。
  3. 实现错误分类、定位、报错信息输出，支持 --recover 恢复模式。
  4. 补全 Javadoc 文档并生成 HTML 文档目录。
  5. 扩展 JUnit 自动化测试与性能比较脚本。

3. 输入语言定义

输入是一行中缀表达式，仅允许单个数字与加减运算符，不允许空白字符。

Expr  -> Term (('+'|'-') Term)*
Term  -> digit
digit -> '0'..'9'

输出为对应后缀表达式；若发生错误，stdout 追加 (error)，并在 stderr 给出错误类别与位置。

4.1 尾递归消除

原始程序在 rest() 中尾递归处理连续运算符。当前版本改为 expectOperand + pendingOp 的循环状态机：先消费数字，再在下一数字出现时回填运算符，实现与原算法一致的后缀输出顺序。

4.2 错误处理策略

类别触发条件示例输出
词法错误 空白字符、非法字符 1 +2 / 1a+2 LEXICAL error at position n
语法错误 缺运算符、缺运算量、空表达式 95+2 / 9-5+-2 SYNTAX error at position n

4.3 错误恢复

默认模式遇首错停止；--recover 模式下会跳过当前错误字符继续扫描，用于一次运行报告更多错误。

11. 实验结论

  1. 实例字段替代静态字段后，Parser 状态边界更清晰，规避多实例共享状态隐患。
  2. 尾递归消除提升了实现可控性，但真实性能需结合完整功能实现进行测量。
  3. 错误分类 + 定位 + 可选恢复显著提升了程序鲁棒性与可诊断性。
  4. 跨平台脚本（sh/bat）使提交结果在不同系统环境中更易复现。

12. 实验心得与反思

12.1 从“能跑”到“可交付”

实验初始版本关注功能最小闭环，工程层面约束很弱。本次将错误模型、测试脚本、文档链路补齐后，项目从课堂 demo 变成了可评估、可复现的提交资产。

12.2 设计权衡

为保留原有输出兼容性，stdout 只保留后缀结果与 (error) 标记，详细错误落到 stderr。该策略在自动对拍与人工调试之间取得平衡。

12.3 后续改进方向

  • 扩展语法到括号、乘除与优先级规则。
  • 引入更系统的 panic-mode 同步机制，提升恢复质量。
  • 把测试与文档生成挂到 CI，形成持续质量门禁。
