本实验的目标不是重写整个软装置，而是在既有 GUI、测试器和异常类基础上，精准实现 calculate(String expression) 的行为，使其同时满足：

  • 能够计算复杂表达式的正确数值结果。
  • 能够区分数值型与布尔型中间结果，避免“看起来能算、其实语义错误”的情况。
  • 在错误输入下抛出和官方测试预期一致的异常类型，而不是笼统失败。
  • 保持实现简单、可读、可复用，不为实验额外引入不必要抽象。

3. 设计与实现

Lexer

词法阶段

负责空白跳过、数字识别、标识符识别和多字符关系运算符识别。数字采用手写扫描逻辑，避免把非法格式留到 Double.parseDouble 之后才暴露。

Parser

语法阶段

使用递归下降逐层处理优先级：conditional → logical → relation → additive → multiplicative → power → unary → primary，实现清晰、调试成本低。

Value

语义阶段

通过显式 Value 类型携带 double 或 boolean，在调用 requireNumber() / requireBoolean() 时完成类型检查并抛出语义异常。

设计上我刻意没有引入 AST 类层次或 Visitor。对本实验而言，目标是尽快对齐实验框架与异常口径；直接在递归下降过程中完成求值，结构更简单，也更符合 KISS 与 YAGNI。

6. 关键代码片段

数值 / 布尔双类型封装

private static final class Value {
    private final boolean booleanType;
    private final double numberValue;
    private final boolean booleanValue;

    private double requireNumber()
        throws TypeMismatchedException {
        if (booleanType) {
            throw new TypeMismatchedException(
                "Numeric value expected.");
        }
        return numberValue;
    }
}

三目运算的类型约束

private Value parseConditional()
    throws ExpressionException {
    Value condition = parseLogicalOr();
    if (lexer.peek().type != TokenType.QUESTION) {
        return condition;
    }

    lexer.next();
    boolean branch = condition.requireBoolean();
    Value trueValue = parseConditionalOperand();
    expect(TokenType.COLON, ...);
    Value falseValue = parseConditionalOperand();
    if (trueValue.booleanType != falseValue.booleanType) {
        throw new TypeMismatchedException(...);
    }
    return branch ? trueValue : falseValue;
}

8. 实验结论

Lab2 的关键不在“把表达式算出来”，而在于让表达式求值器同时具备编译器前端的三个基本特征：明确的词法边界、稳定的语法优先级和可解释的语义错误。

本次实现用一个文件完成了这些能力，但仍保持了职责分离：Lexer 只负责 token，Parser 只负责优先级和结构，Value 只负责类型约束。最终结果是：

  • 官方标准测试 16 / 16 全通过。
  • 异常类型与实验框架完全对齐，没有警告与误报。
  • 提交物完整，包括源码、Shell 脚本、Javadoc、HTML 报告与 PDF 报告。
