## 1. Oberon-0 词汇表

### 1.1 保留字 (Reserved Words)

保留字是语法结构的一部分，不能被程序员声明为标识符。

| 分类 | 保留字 | 数量 |
|------|--------|------|
| 模块结构 | MODULE, BEGIN, END | 3 |
| 声明 | CONST, TYPE, VAR, PROCEDURE | 4 |
| 控制流 | IF, THEN, ELSIF, ELSE, WHILE, DO | 6 |
| 类型构造 | ARRAY, OF, RECORD | 3 |
| 运算符 | DIV, MOD, OR | 3 |

共 **19** 个保留字。

### 1.2 关键字 (Keywords — 预定义标识符)

关键字是预定义标识符，遵循标识符的作用域规则，可被重新声明覆盖。

| 分类 | 关键字 | 数量 |
|------|--------|------|
| 预定义类型 | INTEGER, BOOLEAN | 2 |
| 布尔值 | TRUE, FALSE | 2 |
| 预定义过程 | READ, WRITE, WRITELN | 3 |

共 **7** 个关键字。

> **保留字与关键字的区别**：保留字在词法层面被识别为独立的 token 类型，即使出现在标识符上下文中也不允许作为标识符使用。关键字本质上是标识符，仅因预定义而具有特殊含义，在作用域规则下可被覆盖。

## 2. 词法规则的正则定义式

```
letter        → [a-zA-Z]
digit         → [0-9]
octal_digit   → [0-7]

identifier    → letter (letter | digit)*
               (长度限制: ≤ 24 字符)

DecimalInteger → [1-9] digit*                (长度限制: ≤ 12 字符)
OctalInteger   → 0 octal_digit*              (长度限制: ≤ 12 字符)
number         → DecimalInteger | OctalInteger

comment       → "(*" ([^*] | "*"+ [^*)])* "*"+ ")"
               (不允许嵌套)

WhiteSpace    → space | tab | newline | carriage_return
```

### 2.1 关键词法约束

1. **大小写无关**：保留字和标识符均不区分大小写（MODULE = Module = module）
2. **整数常量与标识符分隔**：整数常量后若紧跟字母，视为非法；必须以空白分隔
3. **八进制约束**：以 0 开头的整数常量只能是八进制（仅允许 0-7），数字 8 和 9 为非法
4. **标识符长度**：不允许超过 24 个字符
5. **整数长度**：无论十进制还是八进制，不超过 12 位数字
6. **注释**：(* 与 *) 配对，不允许嵌套

## 3. Oberon-0 与其他语言的词法规则比较

| 特性 | Oberon-0 | Pascal | C/C++/Java |
|------|----------|--------|------------|
| 大小写敏感性 | 不敏感 | 不敏感 | 敏感 |
| 注释语法 | `(* *)` | `(* *)` 或 `{ }` | `/* */` 或 `//` |
| 整数进制 | 十进制 + 八进制 | 十进制 + 十六进制 | 十进制 + 八进制 + 十六进制 |
| 标识符长度限制 | ≤ 24 字符 | 通常无 | 通常无 |
| 整数长度限制 | ≤ 12 数字 | 通常无 | 通常无 |
| 布尔字面量 | 无（TRUE/FALSE为标识符） | 有 TRUE/FALSE | 有 true/false |
| 实数/浮点数 | 不支持 | 支持 | 支持 |
| 字符串类型 | 不支持 | 支持 | 支持 |
| 字符类型 | 不支持 | 支持 | 支持 |

### 关键差异分析

1. **大小写无关**避免了 C/Java 中 `if` vs `If` 的歧义，也简化了词法分析中的符号表管理
2. **无浮点数**避免了小数点与域选择符 `.` 的歧义，不需要复杂的上下文判断
3. **无字符串/字符**避免了转义序列处理的复杂性
4. **注释不与除法冲突**：Oberon-0 的注释使用 `(* *)` 而非 `/* */`，与乘法运算符 `*` 完全区分
5. **严格的分隔规则**：整数后必须有空白才能跟标识符，这是 Pascal 家族的语言特性，避免了 C 中 `25id` 这类歧义

## 4. JFlex / JLex / GNU Flex 差异比较

| 特性 | JFlex | JLex | GNU Flex |
|------|-------|------|----------|
| 实现语言 | Java | Java | C |
| 生成目标语言 | Java | Java | C |
| 速度 | 快（DFA 优化好） | 中等 | 最快 |
| 正则语法 | 丰富（Unicode、预定义类） | 基本 | 丰富（POSIX 字符类） |
| 维护状态 | 活跃（最新 1.8.2, 2020） | 停止维护 | 活跃 |
| 平台支持 | 跨平台（JVM） | 跨平台（JVM） | Unix/Linux 为主 |
| %cup 集成 | 内置支持（`%cup` 指令） | 需手动配置 | 需手动配置 |
| 输入格式 | `.flex` 文件 | `.lex` 文件 | `.l` 文件 |
| 规则语法 | `pattern { action }` | `pattern { action }` | `pattern { action }` |

### JFlex 的优势

1. **更好的 DFA 优化**：JFlex 生成的 DFA 表通常比 JLex 更紧凑
2. **Unicode 支持**：天然支持 Unicode 字符类别（`\p{Alpha}`等）
3. **JavaCUP 集成**：通过 `%cup` 指令自动生成与 CUP 兼容的 Symbol 对象
4. **活跃维护**：JFlex 持续更新，有完整的用户文档

### JFlex 与 GNU Flex 的关键差异

- **语言生态**：JFlex 面向 Java 生态，GNU Flex 面向 C/C++ 生态
- **集成方式**：JFlex + JavaCUP = GNU Flex + GNU Bison（yacc）
- **性能**：GNU Flex 生成的 C 代码通常更快，但 JFlex 的 Java 实现在 JVM 优化下也有不错表现
- **状态处理**：两者都支持词法状态（start conditions），但语法细节不同
